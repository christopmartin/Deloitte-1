What Is This Tool?
The Agentic Workbench is a design-intelligence platform for ServiceNow implementations — it turns messy documents and live SN instances into governed, AI-generated application designs.
🤖 AI Ingestion
✍️ Design Review & Governance
🔄 ServiceNow Round-Trip

🤖 AI Ingestion

Upload existing documents — BRDs, process maps, meeting notes, SN config exports — and Claude reads them, extracts structured design entities (use cases, workflows, agents, tools, requirements, test cases), and populates a design database.

Think of it as "AI fills in the blanks from your documents." Analysts upload source material; the AI does the first-pass design work.

✍️ Design Review & Governance

A structured workspace for reviewing, editing, and governing the AI-generated design. Supports change packets (tracked change proposals with approval workflow), baselines (point-in-time snapshots), Now Assist cost estimation, and AI-powered quality review.

Every change is tracked. Every AI action is audited.

🔄 ServiceNow Round-Trip

⚠️ Partial

Connect to a live ServiceNow instance, reverse-engineer its current configuration into the Workbench, reconcile it against the design, and (eventually) write approved changes back to SN. This is the highest-value capability. Ingest + reconcile are working; write-back is the main remaining gap.

Who uses it?

Deloitte consultants on ServiceNow engagements — primarily architects and analysts who need to design, document, and govern SN applications at scale across large, complex client environments.

⚠️
Current production readiness: 4 / 10. The AI integration and SN ingestion pipeline are solid. Authentication does not exist (just an HTTP header), multi-tenancy is modeled but not enforced, and there is no test suite or CI/CD. The production roadmap addresses all of this — see docs/PRODUCTION_WORKPLAN.md.

Three-tier system: Vanilla JS SPA → Express API → SQLite, with Anthropic Claude and ServiceNow as external services.

The SPA shell that bootstraps the entire application. On load it reads asdlc_user_id and asdlc_project_id from localStorage, renders the sidebar nav, and delegates to module render(container) functions on nav click.

🤖 AI Ingestion ✅ Working

▼
frontend/modules/ingest.js

The primary AI data-entry mechanism. Upload source documents; Claude extracts structured design entities and stages them for review before promotion to the live design.

  • Upload DOCX, TXT, CSV files (multer, 30 MB limit)
  • AI extraction pipeline: Sonnet extracts, Opus synthesizes cross-document
  • Staged review of extracted entities before promotion
  • Clarification Q&A: AI asks follow-ups when source is ambiguous
  • User feedback loop (thumbs up/down on individual extractions)
  • Full ingest history with status tracking

Known gaps:

  • #36 RASIC fields not consistently populated — recurring AI regression
  • #40 Clarification auto-apply overwrites cost params project-wide instead of scoping to the document
  • Stealth If ANTHROPIC_API_KEY is missing, silently uses regex heuristics instead of Claude — no warning shown

Key Patterns

Schema conventions

UUID primary keys — TEXT, generated by db.generateId().

Slugs — human-readable IDs (UC-001, WF-002) generated by db.nextSlug() per project.

JSON in TEXT columns — goals, fields, relationships stored as JSON strings. Migrating to jsonb in PostgreSQL.

Soft delete — lifecycle_status: active / archived / deprecated / retired.

Full audit trail — db.auditLog() called on every write; stores before/after state in asdlc_audit_log.

ServiceNow provenance

SN-sourced entities carry five columns that power the round-trip identity:

source_system — always "ServiceNow"

source_sys_id — the SN record's UUID

source_table — SN table name

source_fluent — normalized Fluent IR for round-trip matching

source_hash — hash of SN state at time of ingest

🔐
Multi-tenancy gap. Every design entity carries project_id. Projects carry client_id. The schema is correct but the API does not yet verify that the authenticated user belongs to the requested project. Any user can access any project's data. This is the primary security gap — fixed in Phase 4 of the production roadmap.

How Prompts Work

Externalized Prompt Templates

Prompts are Markdown files in backend-node/agent/prompts/*.md, loaded at runtime by prompt-templates.js. Prompt changes don't require code deployments — just edit the .md file.

The ingest loop injects a small SN context block into every prompt, plus a read_wiki_page tool that lets Claude pull detailed SN design reference on demand (progressive disclosure — avoids loading 33K tokens unless needed).

Extended Thinking

Opus calls for reasoning-heavy tasks (reconciliation, synthesis) use Anthropic's extended thinking (adaptive mode) when enabled.

This adds latency but significantly improves quality on complex multi-document design generation tasks. Configurable per task type.

Usage Tracking

📊
Every Claude API call is logged to asdlc_ai_usage with: input tokens, output tokens, estimated cost, model used, and the role that triggered it. Viewable in Admin → AI Settings. Per-tenant cost budgeting (alerting when spend exceeds threshold) is on the production roadmap.
ServiceNow Integration
Per-project encrypted connections, Fluent-notation round-trip identity, and a partially-complete write-back pipeline.

🔑 Connection Model

Each project stores its own SN connection: instance URL, username, and AES-256-GCM encrypted password in asdlc_project.sn_password_enc.

Encryption key: ASDLC_ENCRYPT_KEY environment variable. If missing, credentials fall back to plaintext storage with a "plain:" prefix.

⚠️
Always set ASDLC_ENCRYPT_KEY — without it, SN passwords are stored in plaintext in the database (issue #76).

🌐 Corporate TLS Issue

Deloitte's SSL inspection proxy causes SELF_SIGNED_CERT_IN_CHAIN errors on SN API calls.

Dev workaround: SN_INSECURE_TLS=true in .env disables certificate verification.

🔒
Not acceptable for production. The production fix is NODE_EXTRA_CA_CERTS=<path-to-deloitte-proxy-cert>.

Fluent Notation — Round-Trip Identity

The round-trip system uses Fluent notation — an internal intermediate representation (IR) that encodes SN design objects in a normalized form. When an entity is ingested from SN, its Fluent representation is stored in source_fluent.

When writing changes back to SN, Fluent is used to map Workbench design entities to the correct SN sys_ids. This is what makes the round-trip reliable rather than fragile name-based matching — even if the SN record name changes, the Fluent notation + sys_id maintains identity.

Six structured sessions to get a new team member fully oriented. Run sessions 1–3 in week one, 4–6 as the developer sets up and starts contributing.
Session 1 · Week 1
Architecture & Data Model
⏱ 2 hours👥 All team

Before this session

  • Read: Architecture and Data Model sections of this guide
  • Open: backend-node/schema.sql — skim the table names
  • Have running: node server.js and confirm the home dashboard loads
  • Bring: any questions from your first scan of the repo

Big picture first — how the pieces fit together and how data is structured.

  • System architecture diagram and request flows
  • Tech stack decisions and why (no framework, no build step)
  • 68-table schema — key entities and relationships
  • Multi-tenancy model (schema vs. enforcement gap)
  • Audit trail and soft-delete patterns
Session 2 · Week 1
Working Features Demo
⏱ 2 hours👥 All team

Before this session

  • Read: Feature Inventory section of this guide — know what to expect
  • Have running: the app with a demo project open
  • Have ready: a sample DOCX or TXT file to upload during the demo
  • Confirm: access to a SN dev instance, or presenter will share screen

Live walkthrough of what works today — run real user journeys end to end.

  • Create project, configure SN connection
  • Upload a document → AI extracts → review staged → promote
  • Review design in Design Review module
  • Create and approve a change packet
  • Run SN assessment on a live instance
  • Admin: AI settings, best practices, audit log
Session 3 · Week 1
Stubs, Gaps & Roadmap
⏱ 1 hour👥 All team

Before this session

  • Read: Known Issues and Deferred (v2) sections of this guide
  • Open: BACKLOG.md in your editor
  • Skim: production work plan — the Critical Path table

Know what's fake before you spend time on it. Context for what's in v1 vs. v2.

  • Walk through every ⚠️ Partial and 🔲 Stub module
  • Explain each stub: why it's fake, what the real version requires
  • Review the 7 critical issues in BACKLOG.md
  • Overview of production roadmap phases
  • v1 vs. v2 scope decision (still open — to be decided in Phase 2)
Session 4 · Week 2
AI Pipeline Deep-Dive
⏱ 2 hours👩‍💻 Developers

Before this session

  • Read: AI Integration section of this guide
  • Open: backend-node/agent/ — note the directory structure
  • Skim: any .md file in backend-node/agent/prompts/
  • Have running: app with browser devtools (Network tab) open

How the AI integration actually works — follow an ingestion call from HTTP to token.

  • Model routing: ai-config.js + asdlc_app_setting
  • Prompt templates: agent/prompts/*.md + prompt-templates.js
  • SN wiki context injection (wiki-context.js)
  • Extended thinking configuration for Opus
  • Usage tracking: asdlc_ai_usage table
  • Fallback behavior when ANTHROPIC_API_KEY is missing
Session 5 · Week 2
ServiceNow Integration
⏱ 1 hour👩‍💻 Developers

Before this session

  • Read: ServiceNow section of this guide
  • Open: backend-node/crypto-util.js
  • Confirm: access to a SN dev instance for the live connection demo

How the SN connection and round-trip identity system work.

  • SN connection config and credential encryption (crypto-util.js)
  • The TLS workaround and production fix
  • Fluent notation — what it is and why it matters
  • Round-trip status: what works vs. what doesn't
  • SN provenance columns on design entities
Session 6 · Week 2
Local Setup & First Contribution
⏱ Half-day👩‍💻 Developers

Before this session

  • Complete: sessions 1–5 first
  • Verify: node --version returns 22.5 or higher
  • Have: .env file with ANTHROPIC_API_KEY set
  • Read: Local Setup section of this guide

Get running and make a real commit — best way to internalize the codebase.

  • Run through setup commands in this guide
  • Explore server.js route structure with a guide
  • Pick one of the critical/high issues from BACKLOG.md
  • Write the fix, test manually, open a PR
  • Q&A on anything unclear from sessions 1–5
