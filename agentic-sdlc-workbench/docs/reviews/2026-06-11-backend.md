## Critical

1. **[correctness] [server.js:6943](../../backend-node/server.js)** — The ingest `promote` handler writes the change-packet header, then loops `insertItem.run()` + `markPromoted.run()` per extraction, then updates the ingest doc — **all without a transaction**. A failure mid-loop leaves a half-populated CP with some extractions flipped to `promoted` and others not, corrupting the ingest's apply state. Other handlers in this file already use `db.exec('BEGIN')`. → Wrap the CP insert + per-item loop + doc update in `BEGIN`/`COMMIT` with `ROLLBACK` on catch.

2. **[correctness] [server.js:7201](../../backend-node/server.js)** — ServiceNow `sync` write-back is non-atomic across multiple statements: the HITL packet insert, the auto packet insert + `approveAndApplyCp`, the `source_hash` advancement loop, and `markSynced` each commit independently. A throw after the HITL packet but before/within auto-apply leaves an orphaned pending CP and partially-advanced hashes, desyncing the next sync's tier-0 diff. → Run the whole materialize block in one transaction so a failed sync rolls back cleanly.

## High

3. **[correctness] server.js (whole file)** — There is **no Express error-handling middleware** `(err,req,res,next)` anywhere (only the 404 catch-alls at [7984](../../backend-node/server.js)/7991). ~30+ synchronous handlers run `db.prepare().get()/run()` with no try/catch (e.g. [296](../../backend-node/server.js), 2390, 2475, 4142, 4774); a thrown SQLite error (FK/CHECK/UNIQUE, bad JSON body) or a rejected async promise auto-forwards to Express 5's default handler → bare 500, no JSON body, UI sees an opaque failure. → Add a terminal `app.use((err,req,res,next)=>res.status(err.status||500).json({error:err.message}))` after all routes.

4. **[security] [server.js:4002](../../backend-node/server.js)** — The `/library` endpoint interpolates the `scope` query param into SQL via `scope.replace(/'/g,"''")` string-building (4 occurrences: 4002, 4020, 4033, 4047) instead of a bound `?`. Manual single-quote escaping is the only thing preventing injection — it's the one non-parameterized query in server.js. → Replace with `AND uc.visibility_scope = ?` and push `scope` into the params array like every other list endpoint.

5. **[correctness] [server.js:6691](../../backend-node/server.js)** — The clarification auto-apply blindly overwrites `runs_per_period` on **every** workflow in the project (`WHERE project_id=? AND (runs_per_period IS NULL OR =0)`) and `cost_model` on every agent, from a single free-text answer parsed with `/\d+/`. One document's clarification silently rewrites volumes project-wide, skewing all cost projections. → Scope the auto-apply to workflows/agents linked to the originating ingest, or surface the parsed value for confirmation before writing.

6. **[security] [server.js:7142](../../backend-node/server.js)** — The SN `sync` endpoint returns raw `err.message` from the live ServiceNow REST pull in its 500/502 bodies (7171, 7234), which can surface instance URLs / credential-hint text to the client. → Scrub or generalize `err.message` in `servicenow/sync` responses; log the detail server-side only.

## Medium

7. **[structure] [server.js:4142](../../backend-node/server.js)** — No service layer: ~167 routes plus heavy business logic live inline in one 8,005-line file. The change-packet materializer (700–1341), the SN/build markdown assemblers (4946–~6500), the promote orchestration (6721–6989), and the SN sync pipeline (7142–7256) are all business logic. → Extract `agent/materializer.js`, `agent/build-export.js`, `agent/promote.js`, `agent/sn-writeback.js`, then split routes into `express.Router()` modules by group (changePackets, projects, costs, workflows, requirements, ingest, designReports).

8. **[structure] [server.js:248](../../backend-node/server.js)** — The fetch-by-id → 404 → `res.json` preamble is duplicated in ~50+ handlers (e.g. 248, 369, 505, 1799, 2382; and the project-or-404 variant ~12× across design reports: 4143, 4339, 4518, 4775, 4947). → Add a `getOr404(res, sql, params, label)` helper and a `getProjectOr404(req,res)` loader.

9. **[performance] [server.js:1130](../../backend-node/server.js)** — `db.prepare()` re-compiled inside per-row loops (~7 hot sites): `generateAndInsertTests` (1134), `materializeRequirementLinks` (1184), `inferLinks` apply (2250), `test-coverage/infer` (3959), and the design-report N+1 at 4290 / 4828 (prepared+run once per agent/UC). → Hoist the prepared statements above their loops (build-export at 5023 already does this correctly — follow that pattern).

10. **[performance] schema.sql (missing indexes)** — The most-joined columns in the design-report/build-export slice have no index: `asdlc_workflow_step(workflow_id)`, `asdlc_workflow_step_cost_binding(workflow_step_id)`, `asdlc_hitl_gate(workflow_id)`, `asdlc_agent_spec(project_id)` & `(workflow_id)`, `asdlc_tool(project_id)`, `asdlc_workflow(use_case_id)`, `asdlc_agent_use_case`, `asdlc_agent_tool`, `asdlc_assist_rate_card(skill_name)`. Every report does per-row lookups on these. → Add the indexes to [schema.sql](../../backend-node/schema.sql).

11. **[performance/AI cost] [server.js:3270](../../backend-node/server.js)** — `/cost-estimate` is the only direct `client.messages.create` in server.js and it (a) hardcodes `model:'claude-opus-4-5'` + `max_tokens:4096` instead of `aiConfig.resolveModel()/getMaxTokens()`, and (b) never writes to `asdlc_ai_usage`, so this Opus call is invisible to the `/usage` cost dashboards and uncapped by the admin model setting. → Route it through `aiConfig` and log `message.usage` (source `'cost_estimate'`). *(Relates to the agent-area pricing finding and backlog #21.)*

12. **[security] [server.js:4642](../../backend-node/server.js)** — Several handlers interpolate a table/column name into SQL from the `RT_DESIGN`/`RT_EXPORT`/`LINK_*_META` lookup maps (e.g. `SELECT * FROM ${spec.table}`, `UPDATE ${spec.table} SET ${setCols}`; reps 4642, 4661, 5151, 7840). Values come from server-defined frozen maps keyed by validated input — not injectable today, but fragile. Row values are correctly parameterized. → Add a hard allowlist assertion (`if(!RT_DESIGN[scope]) return 400`) at each entry and document that table/column names must only ever come from these maps. *(Same invariant as the agent-area `entity-registry` finding.)*

13. **[correctness] [db.js:187](../../backend-node/db.js)** — The `MIGRATIONS` loop swallows **all** errors (`catch {}` at ~298), so a genuinely broken `ALTER`/`CREATE TABLE` is hidden as if it were a benign duplicate-column. It also contains a data `UPDATE` (`cp_origin` backfill, 187–190) whose failure would be invisible. → Only swallow when `err.message` matches "duplicate column"/"already exists"; log everything else; move data `UPDATE`s into a logged `DATA_MIGRATIONS` path.

14. **[correctness] [server.js:553](../../backend-node/server.js)** — The per-item approve/reject handlers run the writes inside `BEGIN`/`COMMIT`, but fetch `item`/`cp` and evaluate `allDecided` eligibility **before** `BEGIN` (555–568). Two concurrent approvals of the last two items of one CP could both pass and double-flip the CP status / double-bump the project version. Low likelihood on a single-user tool. → Move the item-status guard inside the transaction (re-read with the write) if concurrency ever appears.

## Top 3 recommended next actions

1. **Wrap the two non-transactional multi-writes** (findings 1 & 2 — ingest `promote` and ServiceNow `sync`) in `BEGIN`/`COMMIT`/`ROLLBACK`. These are the core write paths of the app and can half-commit, corrupting ingest/sync state.
2. **Add the terminal error-handling middleware** (finding 3). It's one `app.use(...)` that converts every uncaught throw across ~167 routes from an opaque 500 into a structured JSON error — and is a prerequisite for cleaning up the mixed error idioms (#18).
3. **Fix the project-wide clarification overwrite** (finding 5) — it silently rewrites `runs_per_period`/`cost_model` across the whole project from one document's answer, producing wrong cost projections with no visible cause.
