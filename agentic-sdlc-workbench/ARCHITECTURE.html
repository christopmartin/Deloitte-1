How It Works — The Core Loop

1

Upload

Delivery team uploads a BRD, requirements transcript, or ServiceNow source (Fluent IR) via the Ingest module.

2

Extract

Claude runs an agentic tool-call loop (up to 20 iterations) extracting use cases, workflows, agents, tools, requirements, guardrails, and data sources. Confidence determines enrichment level.

3

Cross-Check

Deterministic + optional LLM tiers scan for design drift and requirement conflicts. Blocking questions pause the flow; FYI notes proceed.

4

Change Packet Review

Staged extractions become a Change Packet. Product owners approve or reject each item individually. Disputed items return to extraction. Quality-reviewer flags missing or inconsistent fields.

5

Apply & Baseline

Approved items are applied atomically to the design tables with a full audit trail. A locked Baseline captures an immutable snapshot (Draft → Build → Pilot → Production).

6

ServiceNow Round-Trip

The approved design is exported as a Fluent deployment plan and pushed to the ServiceNow instance. sys_id provenance is written back to enable the next sync cycle.

Frontend Modules

20 Vanilla JS ES modules loaded on demand by the SPA shell (app.js). Each exports a render(container) function. User identity and project context are stored in localStorage and sent as X-User-ID on every API call.

Key Architectural Decisions

Ten decisions that shaped the current design. Understanding these will help you avoid re-litigating settled trade-offs and know which assumptions are safe to change.

1

Slug Naming Convention

Every entity gets a stable human-readable slug: UC-001, WF-002, AG-003, T-004, S-005, FR-006, NFR-007, AC-008, TC-009, DM-010, etc. Slugs are per-project (except global tools which have NULL project_id). They survive in prompts, audit logs, requirements docs, and CP item references — UUID alone is too opaque for cross-references in Claude conversations.

2

Supervision Model Enum

Three values: Advisory-only, Supervised HITL, Autonomous. The quality-reviewer validates required fields per supervision model (e.g., Autonomous requires more complete agent specs than Advisory-only). The old Assisted value was migrated to Supervised HITL — do not re-introduce it.

3

Per-Item Change Packet Decisions

Product owners can approve or reject each item in a CP individually (item_status: pending / approved / rejected). This allows partial CPs to proceed while a disputed item goes back to the extraction loop. Whole-CP approval is a convenience shortcut that approves all pending items atomically.

4

Enrichment Levels

faithful — transcribe only what is explicitly stated. balanced — fill logically implied fields (default, used for most documents). suggestive — Claude proposes net-new best-practice items (system_generated = 1 flag set). Controls how liberally Claude extends beyond the source material.

5

Cross-Check Tiers (Plan C)

Tier 0 (always free): deterministic salient-token drift scan — extract key identifiers from changed entities and scan the design corpus for references. Tier 1 (bounded LLM): check new/changed extractions against all FR/NFR. Tier 2 (conditional): deep scan + new clarifications, only when Tier 0 or 1 found something with non-trivial severity. Never skip Tier 0.

6

ServiceNow Provenance (Level-2)

Level-1 design rows carry source_system, source_sys_id, source_table, source_scope, source_fluent, and source_hash. These are hidden from the business user but critical for round-trip identity: sync → extract → reconcile → deploy → register-sysid writes source_sys_id back, enabling the next sync cycle to detect changes vs the same records. CPs with cp_origin = sn_inbound are never exported back to SN.

7

Per-Project Cost Parameters

Cost overrides live on asdlc_project (cost_per_assist, overage_rate, planning_period, entitlement) — not the global asdlc_cost_assumption singleton. Different clients negotiate different ServiceNow price sheets. The API resolves from project row first, then falls back to the global singleton for safety. The global singleton is now legacy — new cost logic should always use per-project params.

8

AI Guidance Platform Scoping

asdlc_best_practice.platform can be servicenow, generic, or any. When building the extraction prompt, ai-config.getActiveBestPractices() filters by the document's platform. This prevents ServiceNow-specific rules from polluting extractions of, say, a generic workflow design. Platform-tagged rules were added alongside the 2026-06-16 hybrid wiki refactor.

9

Cryptographic SN Credentials

Per-project SN username and password are stored AES-256-GCM encrypted in asdlc_app_setting via crypto-util.js. Falls back to env vars SN_USER / SN_PASSWORD if per-project credentials are not set. The scrubProject() helper in server.js removes the encrypted password from any API response, replacing it with a boolean has_sn_password flag.

10

Two-Layer Design Model

Level-1 = business design in plain English (shown in the workbench UI, editable by humans). Level-2 = SN sys_id + Fluent notation + source hash (never shown in UI, used only by the round-trip engine). This separation means the workbench can be used as a pure design tool without any SN connection, and the SN integration is additive — it writes Level-2 provenance without touching the Level-1 human-facing content.

Common Headers & Conventions

ConventionDetail
X-User-ID headerRequired on every mutating call. Set by app.js:apiFetch() from localStorage.asdlc_user_id.
Async endpointsReturn { status: 'processing', id }. Poll GET .../status or the parent entity until status changes.
Error format{ error: "message", details: {...} } with appropriate HTTP status code.
JSON columnsReturned as parsed objects in GET responses. Send as objects in POST/PUT — server calls JSON.stringify() before write.
PaginationNot implemented — all list endpoints return full result sets. Add ?limit=&offset= when result sets grow large.
CORSAll origins allowed. Headers: GET, POST, PUT, DELETE, OPTIONS + X-User-ID.
