Agentic SDLC Workbench
Repository Database Design — Merged v1.0
Synthesizes prior database-design drafts. Adds Documented/Inferred tagging, a Required column, source-spec citations, and pulls cross-cutting attributes out once.
1. How to read this document
This document defines a logical database design for the Agentic SDLC Workbench repository. It synthesizes two prior drafts: one organized strictly around the spec's §9 entity list, and one with a fuller table inventory derived by extension. The merged version keeps the broader inventory and adds rigor to call out which design decisions are documented in the source spec and which are inferred.
References to the source spec ("Agentic SDLC Workbench Future-State Design v0.2") use § notation throughout — e.g. §9, §A.5.
Source-tagging legend (Src column on every field table)
D  Documented — the field name, value, or concept appears in the source spec.
I  Inferred — added to make the schema implementable. Treat as design proposals open to review.
Required column legend
Y required.   N optional.   C conditionally required (rule explained in Notes).
Caveat: this is a logical model, not authoritative DDL. Field names are illustrative, types are best-fit guesses, and a few questions remain unresolved (see §13). Treat this document as input for a schema-review session, not the final schema.
2. Design Principles
Principle
Source
The atomic unit of truth is the field value.
§3
Every meaningful field value should be traceable to evidence: a source document or transcript with date/time.
§3
The repository is internal only. Clients receive reports and exports, not direct repository access.
§3
Agents propose changes; humans approve through Change Packets with field-level audit underneath.
§3, §6
Project separation is required. Reusable records have explicit scope: Project-only, Client-only, All-clients.
§10
Field-level audit is retained but hidden from normal users unless needed.
§3, §9
The trust dial is per project, per agent, set manually 1-5.
§8
Baselines provide human-readable design snapshots: Draft, Build, optional Pilot, Production, Post-Prod vX.Y.
§7
Hybrid field model: high-use fields are typed columns on entity tables; all fields also exist in asdlc_field_value for source/audit/baseline truth.
Inferred — see §13 Open Q. 2
3. Naming Convention
Tables use the prefix asdlc_ for Agentic SDLC. The prefix and identifiers are a logical naming convention — they can be adapted when the repository is implemented in ServiceNow, an external database, or a hybrid architecture.
Convention
Recommendation
Primary key
UUID. Field named <entity>_id, e.g. project_id, change_packet_id.
Foreign key
Field named <referenced_entity>_id. The FK column on each entity table identifies the referenced table.
Scope field
visibility_scope with values PROJECT, CLIENT, ALL_CLIENTS.
Status fields
Use enums (controlled values), not free text.
Audit fields
created_by, created_at, updated_by, updated_at on every table — pulled out once in §4 below.
Soft delete
lifecycle_status (draft / active / under_review / deprecated / archived). No hard deletes for design records.
Polymorphic ref
Use (scope_type, scope_id) pair where one table can reference multiple entity types (Cost Estimate, Governance Control, Test Scenario, Field Value, Field Audit, Exception, Validation Result, Baseline Item).

8. Trust Dial Model
The trust dial is set per (project, agent) in asdlc_project_agent_setting. Levels are documented in §8 of the spec.
Level
Meaning
Behavior
1
Low trust
Agent drafts only. All proposed changes require human review.
2
Developing trust
Agent prepares Change Packets. Most changes require human approval.
3
Moderate trust
Agent may group low-risk changes. Human approval remains the default for approved baselines.
4
High trust
Agent can apply low-risk draft changes with audit; approved baseline changes still require approval.
5
Very high trust
Agent can apply low-risk changes under defined rules. Exceptions, conflicts, high-risk fields, and baselines still route to humans.
9. Scope and Reuse Model
The visibility_scope cross-cutting attribute (§4) implements the access model from §10. All-client patterns are owned outside the system (§10) but tracked here via asdlc_functional_owner for traceability.
Reuse scope
Meaning
Access / control
PROJECT
Visible only inside one project.
Default for client-specific design data.
CLIENT
Reusable across projects for the same client.
Requires client-specific governance by the internal client team.
ALL_CLIENTS
Reusable method pattern across clients.
Maintained outside this system by Functional Owners. Must not contain proprietary client assumptions.

Implementation guidance for whichever physical store the repository ends up in. Required for performance at scale; optional for early MVP.
• Index every (project_id) column. Most queries filter by project.
• Index (entity_type, entity_id) on asdlc_field_value, asdlc_field_value_audit, asdlc_baseline_item, asdlc_test_scenario (linked_*), asdlc_governance_control, asdlc_cost_estimate, asdlc_change_packet_item, asdlc_validation_result, asdlc_exception. These are the polymorphic-FK lookup keys.
• Index (visibility_scope) on every reusable-pattern table to support fast filtering in the Pattern Library UI.
• Index (status) and (lifecycle_status) on asdlc_change_packet, asdlc_exception, asdlc_agent_task — these drive queue UIs.
• Composite index (project_id, baseline_id) on asdlc_field_value for baseline reports.
• Composite index (workbench_agent_id, started_at DESC) on asdlc_agent_run for agent-activity views.
• Unique constraint on asdlc_field_value (entity_type, entity_id, field_path) — only one active value per field.
• Unique constraint on asdlc_project_agent_setting (project_id, workbench_agent_id) — one trust setting per (project, agent).
• Unique constraint on asdlc_change_packet (packet_code) — human-readable IDs cannot collide.
• Foreign-key constraints with ON DELETE RESTRICT on every reference. No cascade deletes — soft-delete only via lifecycle_status.
• Check constraint: trust_level BETWEEN 1 AND 5.
• Check constraint: when visibility_scope = PROJECT, project_id must be non-null. When ALL_CLIENTS, functional_owner_id must be non-null.

11. Suggestions to make the design more robust
• Create a formal field dictionary listing each field, owner, allowed values, required-by-supervision rules, and governance-sensitive flag.
• Define risk tiers for Change Packets so low-risk edits can eventually be approved more quickly than high-risk ones. The asdlc_change_packet.risk_level column is the hook.
• Generate a Design Change Summary report automatically for every approved Change Packet (the asdlc_report_export.report_type = change_summary path).
• Generate Baseline Comparison reports so teams can easily see what changed Draft → Build → Pilot → Production → Post-Prod.
• Use the trust dial cautiously. Start most agents at levels 1-3 and increase only after successful project experience.
• Add automated duplicate detection on asdlc_tool, asdlc_guardrail, asdlc_test_scenario, asdlc_agent_spec — avoid creating multiple versions of the same reusable pattern.
• Add data-leakage checks before any record is promoted from PROJECT to CLIENT or ALL_CLIENTS scope.
• Use the asdlc_field_value.governance_sensitive flag to mask values from client-facing reports automatically.
• Build a repository health dashboard (the §15 Admin Home view): open Change Packets, failed validations, stale reusable records, missing evidence, unapproved baselines, trust-level exceptions.
• Keep client-facing outputs simple and report-based. Do not expose the repository UI to clients in v1.
• Treat the design repository as a product. Assign a product owner, data steward/admin, and Functional Owners for reusable patterns.

13. Open Questions
Items where the source spec is silent, ambiguous, or contains an apparent gap. Resolve items 1-3 before DDL is finalized; items 4-10 can be deferred to a v1.1 review.
Question
Why it matters
Recommendation
1. Implementation location: ServiceNow-native, external workbench, or hybrid?
Affects security model, integrations, UI, agent orchestration, and client delivery model.
Keep open. Decide after MVP-1 scope and ServiceNow app feasibility are confirmed.
2. Hybrid field model: how many fields are physical typed columns vs. only in asdlc_field_value?
Columns improve reporting and queryability; generic field values improve audit and flexibility.
Use hybrid: keep typed columns on entity tables for high-use fields AND insert/update a matching asdlc_field_value row for source/audit/baseline truth.
3. Polymorphic FK pattern: (scope_type, scope_id) vs. explicit nullable FKs to each candidate.
Affects queryability and referential integrity. Cost Estimate, Governance Control, Test Scenario, Field Value, Field Audit, Exception, Validation Result, and Baseline Item all need this.
Recommend (scope_type, scope_id) pair for compactness; add a check constraint or trigger to enforce that scope_id exists in the table named by scope_type.
4. Is User a first-class entity inside the workbench, or a reference to an external IdP?
Affects whether asdlc_project_member.user_id is a UUID FK or a reference string.
Recommend external IdP reference (Entra ID / ServiceNow user table) — workbench does not own identity.
5. Where do all-client patterns physically live?
§10 says all-client patterns are "maintained outside this system" — could mean separate tables or same tables with project_id = NULL.
Same tables, project_id NULL, visibility_scope = ALL_CLIENTS, functional_owner_id required. Simpler than separate tables.
6. Knowledge Article retrieval: is RAG (vector search) required?
Affects whether asdlc_knowledge_article.embedding_vector is populated and refreshed.
Defer to MVP 4 when KAs become operationally important. Cosmetic field in MVP 1.
7. Trust dial overrides: is the override per-(project, agent) only, or per-(project, agent, behavior_class)?
Affects whether asdlc_project_agent_setting needs additional rows per behavior class.
Per-(project, agent) for v1. Add behavior-class granularity only if v1 experience shows it is needed.
8. Reuse boundary on Project.inherit_from_project_id: do inherited records appear with parent IDs or get re-IDed?
Has implications for how Field Audit reads back across project boundaries.
Inherited records remain parent-owned and read-only from the child project. Field Audit reads cross-project for inherited records.
9. How rich should asdlc_source_reference be in v1?
§3 says source link + date/time is sufficient for v1; full citation (timestamp, page, line, confidence) adds robustness for disputes.
Build the table in MVP 1 but only require evidence_source_id and source_datetime; promote richer citation fields to required in MVP 4.
10. asdlc_orchestration_event scope and retention.
Could grow large. Not strictly required if simpler logging suffices.
Optional in v1. Defer to MVP 5 when operate-phase analytics are needed.
